I. Câu Hỏi Về Kiến Trúc Hệ Thống

1 Kiến trúc tổng quan hệ thống Cơ bản KHOA
Hãy mô tả kiến trúc tổng quan của hệ thống LUMA. Tại sao nhóm chọn kiến trúc này thay vì các kiến trúc khác như Microservices?
Gợi ý trả lời:
  • Nêu rõ các thành phần: Backend (Spring Boot), Mobile (React Native), Admin/Organiser Portal (React.js)
  • Giải thích vì sao chọn Monolithic cho backend (quy mô nhóm nhỏ, thời gian phát triển ngắn)
  • Đề cập đến việc giao tiếp qua RESTful API và WebSocket
Câu trả lời mẫu:

Hệ thống LUMA được xây dựng theo kiến trúc Monolithic layered architecture với 3 thành phần chính:

  • Backend: Spring Boot với cấu trúc 3 tầng (Controller - Service - Repository), sử dụng PostgreSQL làm database chính
  • Mobile App: React Native cho cross-platform (iOS & Android)
  • Web Portals: React.js cho Admin Portal và Organiser Portal

Lý do chọn Monolithic thay vì Microservices:

  • Nhóm có 5 thành viên, quy mô nhỏ phù hợp với Monolithic
  • Thời gian phát triển ngắn (1 học kỳ), Microservices tốn nhiều thời gian setup infrastructure
  • Dễ debug, deploy và maintain hơn trong giai đoạn đầu
  • Có thể migrate sang Microservices sau khi scale nếu cần
Câu hỏi chất vấn thêm:

"Nếu sau này hệ thống có 1 triệu người dùng, nhóm sẽ tách module nào ra thành microservice đầu tiên và tại sao?"

2 Thiết kế Database Trung bình DUY
Hãy mô tả thiết kế database của hệ thống. Các bảng chính là gì? Mối quan hệ giữa chúng như thế nào? Tại sao chọn PostgreSQL?
Gợi ý trả lời:
  • Liệt kê các bảng chính: Users, Events, Registrations, Tickets, Payments, Categories, Cities...
  • Giải thích các mối quan hệ 1-N, N-N
  • Nêu ưu điểm của PostgreSQL: JSONB support, full-text search, performance
Câu trả lời mẫu:

Các bảng chính trong database:

  • users: Lưu thông tin người dùng, vai trò (USER, ORGANISER, ADMIN)
  • events: Thông tin sự kiện, liên kết với organiser (FK user_id)
  • ticket_types: Các loại vé của mỗi sự kiện (1 event - N ticket_types)
  • registrations: Đăng ký tham dự (N-N giữa users và events)
  • payments: Giao dịch thanh toán, liên kết với registration
  • certificates: Chứng chỉ cấp cho người tham dự
  • reviews: Đánh giá sự kiện từ người dùng

Tại sao chọn PostgreSQL:

  • Hỗ trợ JSONB để lưu form fields động của sự kiện
  • Full-text search tốt cho tìm kiếm sự kiện
  • ACID compliance đảm bảo tính toàn vẹn cho giao dịch thanh toán
  • Miễn phí và có cộng đồng hỗ trợ lớn
Câu hỏi chất vấn thêm:

"Khi có 10,000 sự kiện và mỗi sự kiện có 1,000 người đăng ký, bảng registrations sẽ có 10 triệu bản ghi. Nhóm đã optimize query như thế nào để tránh slow query?"

3 Real-time Communication Trung bình KHOA/HUY
Hệ thống có tính năng Chat real-time giữa người dùng và người tổ chức. Nhóm đã implement như thế nào? Tại sao chọn công nghệ đó?
Gợi ý trả lời:
  • Giải thích WebSocket và cách hoạt động
  • So sánh WebSocket vs HTTP Polling vs Server-Sent Events
  • Mô tả cách tích hợp với Spring Boot (STOMP) và React Native
Câu trả lời mẫu:

Chúng em sử dụng WebSocket với STOMP protocol cho tính năng chat real-time:

  • Backend: Spring Boot WebSocket với STOMP (Simple Text Oriented Messaging Protocol)
  • Mobile: Thư viện socket.io-client hoặc native WebSocket trong React Native

Tại sao chọn WebSocket thay vì HTTP Polling:

  • WebSocket duy trì kết nối liên tục, giảm latency
  • Tiết kiệm bandwidth so với polling liên tục
  • Bi-directional communication phù hợp cho chat

Flow hoạt động: User gửi message → WebSocket server → Lưu DB → Broadcast đến recipient qua subscription channel

Câu hỏi chất vấn thêm:

"Nếu người dùng đang offline khi có tin nhắn mới, làm sao để họ nhận được tin nhắn khi online lại? Nhóm xử lý message persistence như thế nào?"

Nhóm thiết kế API theo chuẩn nào? Hãy cho ví dụ về một số endpoint chính và giải thích cách xử lý các HTTP methods khác nhau.
Gợi ý trả lời:

  • Đề cập đến versioning, pagination, error handling
Câu trả lời mẫu:

Chúng em thiết kế API theo chuẩn RESTful:

Ví dụ các endpoints cho Events:

  • HTTP status code chuẩn: 200, 201, 400, 401, 403, 404, 500
  • Pagination với query params: ?page=1&size=20
  • API versioning: /api/v1/...

II. Câu Hỏi Về Backend & API

6 Authentication & Authorization Nâng cao KHOA
Hệ thống có 3 loại user: User, Organiser, Admin với các quyền khác nhau. Hãy giải thích cách nhóm implement Authentication (xác thực) và Authorization (phân quyền).
Gợi ý trả lời:
  • Giải thích JWT token flow: Login → Generate token → Verify token
  • Phân biệt Authentication vs Authorization
  • Mô tả OAuth2 Google login flow
  • Role-based access control với Spring Security
Câu trả lời mẫu:

Authentication (Xác thực):

  • Sử dụng JWT (JSON Web Token) cho stateless authentication
  • Khi login thành công, server trả về access_token (15 phút) và refresh_token (7 ngày)
  • Token chứa user_id, role, expiration time
  • Hỗ trợ OAuth2 Google Login: User đăng nhập Google → Nhận authorization code → Exchange lấy user info → Tạo/link account trong DB → Trả JWT

Authorization (Phân quyền):

  • Sử dụng Spring Security với annotation @PreAuthorize
  • 3 roles: ROLE_USER, ROLE_ORGANISER, ROLE_ADMIN
  • Ví dụ: @PreAuthorize("hasRole('ORGANISER')") trên endpoint tạo sự kiện
  • Resource-level authorization: Organiser chỉ được sửa/xóa event của chính mình
Câu hỏi chất vấn thêm:

"JWT được lưu ở đâu trên client? Nếu hacker lấy được JWT token, họ có thể làm gì và nhóm đã có biện pháp gì để giảm thiểu rủi ro?"

7 QR Code Check-in System Trung bình DUY
Tính năng check-in bằng QR code hoạt động như thế nào? Làm sao đảm bảo mỗi vé chỉ được check-in một lần?
Gợi ý trả lời:
  • Giải thích nội dung trong QR code (ticket_id, hash)
  • Flow quét QR: Scan → Gửi API → Verify → Update status
  • Xử lý concurrent check-in (2 người quét cùng lúc)
Câu trả lời mẫu:

Cấu trúc QR Code:

  • QR code chứa một chuỗi unique: {ticket_id}_{hash}
  • Hash được tạo từ ticket_id + secret key, dùng để verify tính hợp lệ

Flow check-in:

  1. Organiser dùng camera quét QR code
  2. Gửi request đến API: POST /api/v1/checkin với QR content
  3. Server verify hash, kiểm tra ticket tồn tại và thuộc đúng event
  4. Kiểm tra status: nếu đã CHECK_IN → báo lỗi "Vé đã được sử dụng"
  5. Nếu hợp lệ → Update status thành CHECKED_IN + timestamp
  6. Trả về thông tin người tham dự để Organiser confirm

Xử lý concurrent: Sử dụng database transaction với optimistic locking để tránh race condition khi 2 người quét cùng lúc

Câu hỏi chất vấn thêm:

"Nếu người dùng chụp ảnh QR code và share cho người khác, làm sao hệ thống phát hiện và ngăn chặn gian lận?"

9 Certificate Generation Trung bình DUY
Hệ thống tự động tạo chứng chỉ cho người tham dự. Hãy mô tả quy trình tạo và verify chứng chỉ.
Gợi ý trả lời:
  • Công nghệ tạo PDF (iText, Apache PDFBox, hoặc HTML to PDF)
  • Template chứng chỉ và cách điền thông tin động
  • Mã xác thực chứng chỉ và cách verify
Câu trả lời mẫu:

Quy trình tạo chứng chỉ:

  1. Sự kiện kết thúc, hệ thống lấy danh sách người đã check-in
  2. Với mỗi người, tạo certificate record trong DB với unique code
  3. Render HTML template với thông tin: tên người nhận, tên sự kiện, ngày, mã chứng chỉ
  4. Convert HTML to PDF sử dụng thư viện Flying Saucer (hoặc iText)
  5. Upload PDF lên Cloudinary và lưu URL
  6. Gửi email với PDF đính kèm

Verify chứng chỉ:

  • Mỗi chứng chỉ có mã unique và QR code
  • QR code link đến trang verify: /certificates/verify/{code}
  • Trang verify hiển thị thông tin chứng chỉ nếu hợp lệ
Câu hỏi chất vấn thêm:

"Nếu người dùng chỉnh sửa file PDF chứng chỉ (đổi tên), làm sao hệ thống phát hiện chứng chỉ đã bị giả mạo?"

10 Scheduled Tasks Trung bình KHOA/DUY
Hệ thống có các tác vụ định kỳ như gửi email nhắc nhở, tạo chứng chỉ, xử lý thanh toán. Nhóm implement scheduled tasks như thế nào?
Gợi ý trả lời:
  • Spring @Scheduled annotation
  • Cron expression cho các task khác nhau
  • Xử lý khi task fail hoặc chạy quá lâu
Câu trả lời mẫu:

Công nghệ: Spring Boot @Scheduled annotation với cron expression

Các scheduled tasks trong hệ thống:

  • Event reminder: Chạy mỗi giờ, tìm sự kiện diễn ra trong 24h tới, gửi email nhắc nhở
  • Certificate generation: Chạy mỗi 6h, tìm sự kiện đã kết thúc chưa có certificate, tạo và gửi
  • Boost expiration: Chạy hàng ngày, kiểm tra và hết hạn các boost đã hết thời gian
  • Cleanup: Xóa các draft event quá 30 ngày

Ví dụ code:

@Scheduled(cron = "0 0 * * * *") // Mỗi giờ
public void sendEventReminders() { ... }

Xử lý lỗi: Wrap trong try-catch, log error, có thể retry hoặc alert admin

Câu hỏi chất vấn thêm:

"Nếu chạy 2 instance của backend (load balancing), scheduled task có chạy 2 lần không? Làm sao tránh duplicate?"

12 Offline Support Nâng cao HUY
Báo cáo có đề cập vé điện tử có thể truy cập khi không có internet. Nhóm đã implement offline support như thế nào?
Gợi ý trả lời:
  • AsyncStorage hoặc SQLite cho local storage
  • Cache strategy: cache vé khi online
  • Sync mechanism khi có lại kết nối
Câu trả lời mẫu:

Công nghệ: AsyncStorage của React Native để lưu dữ liệu locally

Cách hoạt động:

  1. Khi user xem vé (có internet), app cache thông tin vé và QR code image vào AsyncStorage
  2. Lưu dưới dạng: tickets_{user_id}: [array of ticket objects]
  3. Khi offline, app hiển thị vé từ cache
  4. QR code được lưu dưới dạng base64 string để render offline

Sync khi online:

  • Khi có internet lại, app tự động fetch data mới từ server
  • So sánh và update cache nếu có thay đổi
  • Hiển thị indicator nếu data cũ hơn 24h
Câu hỏi chất vấn thêm:

"Nếu vé đã bị hủy (refund) nhưng user đang offline và không biết, họ vẫn có thể show QR code. Làm sao xử lý tình huống này?"

14 Form Handling & Validation Trung bình KHÁNH/TRINH
Organiser Portal có form tạo sự kiện phức tạp với nhiều fields. Nhóm xử lý form validation như thế nào ở cả frontend và backend?
Gợi ý trả lời:
  • Frontend validation: React Hook Form, Formik, hoặc Yup
  • Backend validation: Spring Validation (@Valid, @NotNull, @Size...)
  • Error message handling và hiển thị
Câu trả lời mẫu:

Frontend (React):

  • Sử dụng React Hook Form để quản lý form state
  • Yup schema cho validation rules
  • Real-time validation khi user nhập
  • Hiển thị error message ngay dưới field bị lỗi

Backend (Spring Boot):

  • DTO class với validation annotations: @NotBlank, @Size, @Email, @Future (cho ngày sự kiện)
  • Custom validator cho logic phức tạp (end_date > start_date)
  • Global exception handler trả về structured error response

Ví dụ validation rules:

  • Event name: required, 5-100 ký tự
  • Start date: required, phải trong tương lai
  • Ticket price: số >= 0
Câu hỏi chất vấn thêm:

"Nếu user disable JavaScript trên browser, frontend validation sẽ không chạy. Backend có đủ validation để chống injection và data corruption không?"

  • Client-side vs Server-side payment processing

  • Xử lý các trường hợp: thành công, thất bại, pending
Câu trả lời mẫu:

Flow thanh toán với Stripe:

  1. User chọn vé và nhấn Pay: Mobile gửi request tạo registration
  2. Backend tạo Payment Intent: Gọi Stripe API tạo payment_intent với amount, currency
  3. Trả client_secret về Mobile: Mobile dùng Stripe SDK hiển thị payment form
  4. User nhập thẻ và confirm: Stripe SDK gửi payment info trực tiếp đến Stripe (không qua server chúng ta - PCI compliance)
  5. Stripe xử lý: Charge thẻ, trả kết quả
  6. Webhook notification: Stripe gọi webhook endpoint của chúng ta với event "payment_intent.succeeded"
  7. Backend xử lý webhook: Update registration status = PAID, tạo ticket, gửi email xác nhận

Xử lý các trường hợp:

  • Success: Tạo ticket, gửi email
  • Failed: Thông báo lỗi, user có thể retry
  • Pending (3D Secure): Đợi user xác thực thêm

  • Different limits cho different endpoints
Câu trả lời mẫu:

  • Sử dụng Bucket4j library tích hợp với Spring Boot
  • Áp dụng sliding window algorithm

Các mức rate limit:

  • Payment records cho tax/audit
  • Aggregate analytics data
Câu hỏi chất vấn thêm:

"User đã là Organiser với 50 events và 10,000 registrations. Nếu họ yêu cầu xóa, có ảnh hưởng gì đến những người đã đăng ký events của họ không?"

33 System Outage Recovery Nâng cao KHOA
Server bị down 2 tiếng đúng lúc có sự kiện lớn đang check-in. Khi khôi phục, làm sao đảm bảo không mất data và check-in hoạt động bình thường?
Gợi ý trả lời:
  • Offline check-in capability
  • Data sync sau khi online
  • Database backup và recovery
  • Incident response procedure
Câu trả lời mẫu:

Prevention:

  • Database automated daily backups
  • Point-in-time recovery enabled
  • Health monitoring và alerting

During outage - Offline check-in:

  • Organiser app có thể cache danh sách registrations
  • Check-in offline mode: validate QR locally, store locally
  • Sync với server khi online lại

Recovery procedure:

  1. Identify root cause (server crash, DB issue, network)
  2. Restore from latest backup nếu cần
  3. Verify data integrity
  4. Sync offline check-ins từ Organiser devices
  5. Resolve conflicts (duplicate check-ins)
  6. Notify affected users về outage và resolution
Câu hỏi chất vấn thêm:

"Nếu 2 Organiser devices check-in cùng một vé khi offline, khi sync sẽ có 2 records. Làm sao resolve conflict này?"

34 Git Workflow Cơ bản Cả nhóm
5 người cùng làm việc trên cùng codebase. Nhóm sử dụng Git workflow như thế nào để tránh conflict và đảm bảo code quality?
Gợi ý trả lời:
  • Branching strategy (Git Flow, GitHub Flow)
  • Pull Request và Code Review process
  • Merge conflict resolution
  • CI/CD integration
Câu trả lời mẫu:

Branching Strategy: GitHub Flow (đơn giản hóa)

  • main: Production-ready code
  • develop: Integration branch
  • feature/xxx: Feature branches
  • bugfix/xxx: Bug fix branches

Workflow:

  1. Tạo feature branch từ develop
  2. Commit thường xuyên với meaningful messages
  3. Tạo Pull Request khi feature hoàn thành
  4. Ít nhất 1 người review và approve
  5. Merge vào develop
  6. Cuối sprint, merge develop → main

Code Review checklist: Logic đúng, không có security issues, coding standards, có tests

Câu hỏi chất vấn thêm:

"Có trường hợp nào 2 người cùng sửa 1 file và gây conflict nghiêm trọng không? Nhóm giải quyết như thế nào?"

35 Testing Strategy Trung bình Cả nhóm
Nhóm có viết tests không? Unit tests, Integration tests hay Manual testing? Tỷ lệ code coverage là bao nhiêu?
Gợi ý trả lời:
  • Types of tests implemented
  • Testing frameworks used
  • What is tested and what is not
  • Test automation
Câu trả lời mẫu:

Testing approach:

  • Unit Tests: JUnit 5 + Mockito cho backend services
  • Integration Tests: @SpringBootTest cho API endpoints
  • Manual Testing: QA cho UI/UX, edge cases

Focus areas:

  • Payment flow (critical - nhiều tests nhất)
  • Authentication/Authorization
  • Registration và Check-in logic

Coverage: ~60% cho backend services (focus on business logic)

Limitations:

  • Mobile app chủ yếu manual test do thời gian hạn chế
  • Không có end-to-end automated tests
Câu hỏi chất vấn thêm:

"Cho em thấy một test case cụ thể mà nhóm đã viết. Giải thích test này kiểm tra gì và tại sao quan trọng."
