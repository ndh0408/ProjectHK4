1Tổng quan đề tài

LUMA là một event marketplace 3-tier (Backend API + Admin Web + Mobile Web) mô phỏng mô hình kinh doanh thực tế của các nền tảng như Luma, Eventbrite, Meetup. Hệ thống phục vụ đồng thời 3 vai trò người dùng: Người tham dự (USER), Nhà tổ chức (ORGANISER) và Quản trị viên (ADMIN).

Bài toán giải quyết

  • Người tham dự: tìm kiếm – đăng ký – thanh toán – check-in – đánh giá sự kiện trên một ứng dụng duy nhất.
  • Nhà tổ chức: tạo sự kiện đa bậc vé (multi-tier ticket), phát hành coupon, thu thập câu hỏi đăng ký, tạo poll khảo sát, xem thống kê funnel, nhận tiền qua Stripe Connect.
  • Quản trị viên: kiểm duyệt sự kiện, phân quyền, giám sát doanh thu, hoa hồng, xử lý báo cáo review.

2Mục tiêu hệ thống

🎯 Mục tiêu nghiệp vụ

  • Xây dựng marketplace 2 phía: Attendee ↔ Organiser.
  • Hỗ trợ vé đa bậc (General/VIP/Early-bird) + QR check-in.
  • Quản lý chiết khấu theo coupon có kiểm soát phạm vi.
  • Tự động hoá: nhắc lịch, cấp chứng chỉ, đóng poll theo giờ.

⚙️ Mục tiêu kỹ thuật

  • Kiến trúc layered rõ ràng (Controller → Service → Repository).
  • API RESTful có Swagger, phân quyền bằng @PreAuthorize.
  • Realtime < 1 giây qua WebSocket STOMP.
  • Thanh toán thật bằng Stripe (test mode) + Webhook idempotent.
  • Thiết kế token hoá (design tokens) và i18n Vi/En cho mobile.

5Danh sách chức năng thực tế (bám source)

🔐 Xác thực & Phân quyền

  • Đăng ký / đăng nhập email + mật khẩu, verify mã OTP qua email, quên mật khẩu.
  • Đăng nhập Google OAuth (POST /api/auth/google).
  • JWT access + refresh token (24h / 7d); interceptor auto-refresh 401.
  • 3 vai trò: ADMIN / ORGANISER / USER chia controller riêng.
  • Rate limit: 10 req/phút cho login, 100 req/phút cho API chung (Bucket4j).

🎫 Sự kiện & Vé

  • CRUD sự kiện với lifecycle: DRAFT → PUBLISHED, Admin APPROVE/REJECT.
  • Multi-tier tickets: mỗi event có nhiều TicketType (tên, giá, tồn kho, vị trí hiển thị).
  • Event recurring (NONE/WEEKLY/MONTHLY/CUSTOM), visibility PUBLIC/PRIVATE, requiresApproval.
  • Category, City, Speaker, Event session (đa phiên).
  • Question tuỳ biến cho form đăng ký + answer lưu về RegistrationAnswer.
  • Bookmark, Follow organiser, Event View tracking (analytics funnel).
  • Ticket Transfer (chuyển vé giữa user).
  • QR check-in (ZXing + mobile_scanner).

💳 Thanh toán & Coupon

  • Stripe PaymentIntent + Checkout Session. Flutter Stripe thu thẻ, không chạm backend.
  • Webhook idempotent: ProcessedWebhookEvent chống xử lý trùng stripeEventId.
  • Refund qua Stripe API khi organiser reject registration đã thanh toán.
  • Stripe Connect cho organiser: link bank, payout, commission.
  • Coupon 2 scope: event-specific (coupon.eventId != null) hoặc organiser-wide (áp dụng mọi event của chính organiser đó). Logic validate tại CouponService.validateCoupon.
  • Coupon có maxUsageCount, maxUsagePerUser, validFrom/Until, minOrderAmount, maxDiscountAmount.

📊 Poll & AI

  • Poll lifecycle: DRAFT → SCHEDULED → ACTIVE → CLOSED (có thể reopen).
  • 3 loại: SINGLE_CHOICE, MULTIPLE_CHOICE, RATING.
  • Auto open khi event bắt đầu / auto close khi event kết thúc / close sau 10 ngày / close khi đủ số vote.
  • Ẩn kết quả tới khi close (hideResultsUntilClosed).
  • Admin Organiser có nút AI-generate poll từ chủ đề — gọi Groq API.
  • Flyway migration V1__Fix_Poll_Boolean_Columns.sql sửa nullable → NOT NULL default.

💬 Realtime & Notification

  • Chat 1-1 + group qua WebSocket STOMP (/ws, topic /topic/chat/{convId}).
  • Authenticate WebSocket bằng JWT ở CONNECT frame.
  • Notification in-app realtime (/topic/notifications/{userId}) + bảng Notification trong DB.
  • Email reminder 24h trước event qua @Scheduled cron job.
  • Email campaign hàng loạt (EmailCampaign + EmailUnsubscribe).

🏢 Organiser Marketplace

  • Profile công khai, logo, website, verification status (admin duyệt).
  • Subscription 3 tier: BASIC/PREMIUM/ENTERPRISE (Stripe subscription).
  • Hoa hồng tự động: OrganiserCommission tính % trên mỗi đơn thành công.
  • Yêu cầu rút tiền (WithdrawalRequest) → admin xử lý.
  • Event Boost 3 gói STARTER/PROFESSIONAL/ELITE → đẩy sự kiện lên feed.

🛡️ Quản trị (Admin)

  • Duyệt event, ban/unban user, đổi role, quản lý organiser.
  • Thống kê doanh thu tổng, breakdown theo organiser/event, funnel analytics.
  • CRUD category, city, boost.
  • Broadcast notification, quản lý review report.
  • Export Excel (POI) / PDF (iText7).

📱 Mobile

  • 15 feature module (auth, home, events, explore, bookmarks, coupons, chat, notifications, profile, calendar, gallery, my_events, questions, create_event, main).
  • Design tokens tách riêng: app_radius, app_shadows, app_spacing, app_typography.
  • Payment screen: chip coupon (ưu đãi cao nhất đầu), hiển thị tổng đã giảm trên nút Pay.
  • i18n Vi/En runtime switch.
  • Map sự kiện (flutter_map), Markdown cho mô tả, PDF viewer cho chứng chỉ.
⚠️ Lưu ý khi thuyết trình: tính năng SeatMap đã bị thay thế bằng Multi-tier Tickets (commit eff1412) — không demo để tránh lỗi. File Events_backup.js ở admin là bản backup khi refactor, không dùng nữa.

🎤 Câu mở đầu

Chào thầy cô và các bạn. Em là Huy, đại diện nhóm 5 bắt đầu phần thuyết trình đồ án LUMA — nền tảng quản lý và bán vé sự kiện. Trong 30 phút tới, 5 thành viên nhóm em sẽ lần lượt trình bày hệ thống theo kiến trúc full-stack từ bảo mật, backend, database, cho đến giao diện admin và ứng dụng mobile.

🎤 Thân bài (tự nói)

Về bài toán: LUMA mô phỏng mô hình của Luma.com, Eventbrite — nơi nhà tổ chức tự đăng sự kiện, phát hành vé đa bậc, chiết khấu bằng coupon; người tham dự tìm – đăng ký – thanh toán – check-in bằng QR; admin kiểm duyệt và quản trị doanh thu. Hệ thống gồm 3 vai trò: USER, ORGANISER, ADMIN — tách hoàn toàn ở tầng API.
Về kiến trúc: Nhóm em chọn mô hình 3-tier. Backend là Spring Boot 3.4.1 chạy Java 21 trên port 8080, cung cấp REST API và WebSocket. Admin là React 18 + Material UI chạy port 3000. Mobile là Flutter compile sang web chạy port 5000 — lý do dùng Flutter web là để demo nhanh trên laptop mà không cần thiết bị thật nhưng vẫn giữ code base mobile-native nếu muốn build APK sau này.
Về bảo mật: Em chọn JWT stateless thay vì session vì có 2 client (admin + mobile) cùng gọi một API — session cookie sẽ phức tạp CORS. Mỗi request có Authorization: Bearer được JwtAuthenticationFilter bóc ra, kiểm tra chữ ký HS256, và set vào SecurityContext. Token access sống 24 giờ, refresh token 7 ngày, auto-refresh khi 401.
Về phân quyền: Mỗi controller endpoint có @PreAuthorize("hasRole('ADMIN')") hoặc hasAnyRole('ORGANISER','ADMIN'). Thư mục controller/ chia làm 4 nhóm admin/, organiser/, user/, pub/ nên không bao giờ có chuyện 1 user truy cập nhầm API admin.
Chống brute-force: Em áp dụng Bucket4j — login giới hạn 10 lần/phút, đăng ký 5 lần/phút. Mật khẩu mã hoá BCrypt 12 round. Ngoài ra còn đăng nhập Google OAuth qua google-api-client.

🎤 Chuyển ý sang Khoa

Đó là tổng quan kiến trúc và bảo mật. Sau khi user đã có token hợp lệ, họ sẽ bước vào giao diện quản trị. Mời bạn Khoa tiếp tục với phần Admin & Organiser Panel.

Admin side: Dashboard hiển thị KPI (tổng doanh thu, user active, event…). Trang Events.js dùng MUI DataGrid có search, filter theo status, phân trang 10/25/50 dòng; nút Approve gọi POST /api/admin/events/{id}/approve, Reject mở dialog nhập lý do. 🎤 Thân bài

Phân tầng: Backend tổ chức 3 lớp theo chuẩn Spring. Lớp controller chỉ nhận request, validate DTO, gọi service. Lớp service chứa business logic, đánh dấu @Transactional để đảm bảo atomic. Lớp repository dùng Spring Data JPA, truy vấn phức tạp dùng @Query JPQL. Exception ném từ service lên sẽ được GlobalExceptionHandler chuyển thành HTTP status phù hợp: BadRequestException → 400, ResourceNotFoundException → 404, UnauthorizedException → 401.
Vòng đời sự kiện: Organiser tạo event ở trạng thái DRAFT, bấm Publish chuyển sang PUBLISHED. Admin duyệt chuyển APPROVED thì event mới xuất hiện trên feed công khai. Mọi xoá đều là soft-delete nhờ annotation Hibernate @SQLDelete — rất quan trọng vì đây là nền tảng tài chính, cần audit log.
Luồng thanh toán — phần em tâm đắc nhất: Khi user mobile chọn vé và bấm Pay, frontend gọi POST /api/user/registrations/{id}/payment-intent. Backend tạo PaymentIntent ở Stripe, lưu stripeClientSecret vào DB và trả về. Flutter Stripe plugin dùng client secret đó thu số thẻ — số thẻ không bao giờ chạm backend của em, đúng chuẩn PCI-DSS. Stripe xử lý xong sẽ gọi webhook POST /api/webhooks/stripe. Em xử lý charge.succeeded: đánh dấu Payment SUCCEEDED, Registration APPROVED, sinh OrganiserCommission record, gửi email có mã QR.
Idempotency: Stripe có thể gửi cùng một webhook nhiều lần. Em lưu stripeEventId vào bảng ProcessedWebhookEvent và kiểm tra trước khi xử lý — nếu đã có thì trả 200 ngay. Đây là best practice bắt buộc với payment gateway.
Coupon 2-scope: Coupon có cột eventId nullable. Nếu khác null — coupon chỉ cho event đó. Nếu null — coupon áp dụng cho mọi event của chính organiser đã tạo coupon. Tức là organiser A không thể tạo coupon để áp dụng vào event của organiser B. Logic này ở CouponService.validateCoupon, commit gần nhất a4d7857. Em cũng check maxUsageCount, maxUsagePerUser, validFrom/Until, minOrderAmount, maxDiscountAmount trước khi áp dụng.
Refund: Khi organiser reject một registration đã trả tiền, service gọi Refund.create() của Stripe, tiền tự hoàn về thẻ, đồng thời đẩy Payment về REFUNDED, giảm usedCount coupon.
Validation: DTO có annotation @NotBlank, @Email, custom validator @StrongPassword (8 ký tự, hoa, số, ký tự đặc biệt), @ValidEventTime đảm bảo endTime > startTime. Sai sẽ trả 400 với list lỗi theo từng field.
