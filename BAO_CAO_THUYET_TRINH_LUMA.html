Smart Waitlist, Live Poll, Funnel Analytics, Networking & Matchmaking (4 nâng cấp); Coupon Engine, Ticket Transfer, Event Comparison, Seat Map, Schedule Builder (5 mới).

Nhóm 4 — Chức Năng AI Smart 5 AI RAG

Đây là điểm nhấn AI của dự án — 5 chức năng AI thực sự áp dụng RAG pattern thay vì chỉ wrapper LLM:

  1. RAG Event Description — Query 5 events thành công nhất cùng category làm reference cho AI học pattern.
  2. RAG Registration Questions — Query 30 câu hỏi từ events tương tự, dedupe, build context.
  3. RAG Event Moderation — So sánh với cả approved + rejected events để ra quyết định có evidence.
  4. AI Smart Pricing — Statistical analysis (median, p25, p75) + price-fill correlation từ 50+ events trước khi gọi AI generate reasoning.
  5. AI Conversational Assistant — Chatbot 2-pass với intent classification + function calling tự động query DB.

❓ Câu Hỏi Từ Ban Giám Khảo

Tại sao chọn JWT thay vì session-based authentication?
JWT là stateless, không cần lưu session trên server, nên scale dễ hơn khi có nhiều người dùng đồng thời. Mobile app cần lưu token để duy trì login giữa các phiên, và JWT phù hợp với REST API. Ngoài ra, JWT cho phép nhiều client cùng dùng chung một backend.
Refresh token được lưu ở đâu? Làm sao đảm bảo an toàn?
Refresh token được lưu trong database với hash, expiry và user_id liên kết. Trên mobile, token được lưu trong Keychain (iOS) hoặc Keystore (Android) thông qua flutter_secure_storage, không lưu vào SharedPreferences thông thường để tránh bị đọc trộm.

AI phân tích và trả về JSON: qualityScore (0-100), recommendation (APPROVE/NEEDS_REVIEW/REJECT), confidence (HIGH/MEDIUM/LOW), strengths, concerns, comparisonInsight ("How does this compare to approved/rejected examples?"), suggestedAction.

Nếu category này từng có nhiều events bị reject vì "thiếu thông tin liên hệ" thì AI sẽ pattern match và cảnh báo. Response có metadata approvedExamplesUsed, rejectedExamplesUsed để admin biết AI đã học từ bao nhiêu case. Quyết định trở nên consistent hơn — không bị phụ thuộc vào kinh nghiệm cá nhân của từng admin.

❓ Câu Hỏi Từ Ban Giám Khảo

RAG Moderation khác gì với version basic ở chức năng trước?
Version basic chỉ gửi event content cho AI và hỏi "is this good?". AI trả lời dựa trên kiến thức training tổng quát của LLM. Version RAG khác ở chỗ: backend query DB tìm 6 events tương tự (3 approved + 3 rejected), build context với cả positive và negative patterns, gửi cho AI cùng với new event. AI có thể đưa ra so sánh cụ thể như "Event này có structure tương tự như Event ABC đã được approve, nhưng lại thiếu thông tin liên hệ giống Event XYZ đã bị reject vì lý do đó". Quyết định có evidence chứ không phải gut feeling.
Nếu category mới hoặc chưa có rejected events thì sao?
Em vẫn fallback gracefully. Response sẽ có approvedExamplesUsed: 0 hoặc rejectedExamplesUsed: 0. AI vẫn trả về recommendation nhưng confidence sẽ thấp hơn — em hướng dẫn AI trong system prompt: "If you have no historical examples, set confidence to LOW and recommend NEEDS_REVIEW".
Có lo bias không? AI có thể học sai pattern từ các quyết định cũ không?
Có. Bias là risk chính của RAG. Nếu admin trước đây reject sai một loạt events, AI sẽ học pattern sai đó. Em mitigate bằng cách: (1) AI luôn explain comparisonInsight để admin verify; (2) Decision cuối cùng vẫn do admin quyết, AI chỉ là gợi ý; (3) Có thể thêm cơ chế "downvote AI suggestion" để retrain prompt sau.

Câu hỏi về Kiến trúc & Công nghệ

Ai trả lời: HUY hoặc bất kỳ ai

Tại sao chọn Spring Boot + Flutter + React thay vì dùng 1 framework duy nhất?
Spring Boot là framework Java mạnh nhất cho enterprise API với ecosystem phong phú (Security, JPA, WebSocket). Flutter cho phép build mobile app cross-platform từ 1 codebase. React với MUI cung cấp dashboard admin chuyên nghiệp. Mỗi tool được chọn vì đúng thế mạnh của nó cho từng use case cụ thể.
Tại sao chọn SQL Server mà không phải MySQL hay PostgreSQL?
SQL Server được chọn vì đây là database mà nhóm đã học trong chương trình, có hỗ trợ tốt cho JPA/Hibernate, và có tính năng enterprise như row-level locking (pessimistic lock) mà em cần cho seat map và waitlist. Ngoài ra, SQL Server có công cụ quản trị SSMS rất trực quan.
Làm sao 3 services giao tiếp với nhau?
Tất cả giao tiếp qua REST API. Mobile và Admin gọi đến Backend qua HTTP (Dio/Axios). Real-time features dùng WebSocket STOMP để push data từ server xuống client. Stripe giao tiếp qua Webhook callback. Không có service-to-service communication trực tiếp giữa Mobile và Admin — cả hai đều đi qua Backend.

Câu hỏi về Bảo mật

Ai trả lời: HUY hoặc TRINH

Hệ thống bảo mật như thế nào? Có những lớp bảo vệ nào?
Nhiều lớp: (1) Authentication bằng JWT + refresh token, password hash BCrypt; (2) Authorization bằng Spring Security với @PreAuthorize trên mỗi endpoint; (3) Rate limiting bằng Bucket4j: 10 request/phút cho login, 5/phút cho register; (4) Validation bằng Spring Validation trên mọi input; (5) CORS configured cho localhost; (6) Stripe xử lý PCI DSS compliance cho payment; (7) Pessimistic locking cho các thao tác concurrent.
Nếu JWT bị đánh cắp thì sao?
Access token có thời hạn 24 giờ nên thiệt hại giới hạn. Refresh token lưu trong database, khi logout sẽ bị xóa. Trên mobile, token lưu trong flutter_secure_storage (Keychain/Keystore), không phải SharedPreferences. Nếu phát hiện bất thường, admin có thể lock user ngay lập tức.

Câu hỏi về Database

Ai trả lời: DUY hoặc KHOA

Dự án có bao nhiêu bảng? Quan hệ chính là gì?
55 entities/bảng. Quan hệ chính: User → Registration (1:N), Event → Registration (1:N), Event → TicketType (1:N), Registration → Payment (1:1), Event → Category (N:1), Event → City (N:1), Event → SeatZone → Seat (1:N:N), User → Conversation → Message (N:N:N). Tất cả được map qua JPA/Hibernate với lazy loading.
Xử lý concurrent access như thế nào? Ví dụ 2 người mua cùng 1 ghế?
Dùng pessimistic write lock ở database level. Khi user chọn ghế, backend chạy SELECT ... FOR UPDATE, lock row đó trong transaction. User thứ 2 sẽ bị block cho đến khi transaction đầu tiên commit hoặc rollback. Ngoài ra, ghế có trạng thái LOCKED với thời hạn 5 phút — nếu không thanh toán thì tự động release.

Câu hỏi về AI/RAG

Ai trả lời: người phụ trách chức năng AI tương ứng

RAG trong dự án này khác gì với RAG chuẩn dùng vector database?
RAG chuẩn dùng vector embeddings + similarity search (Pinecone, Weaviate). Dự án em dùng structured RAG — query SQL database với điều kiện logic (cùng category, top rated, recent) thay vì semantic search. Ưu điểm: đơn giản hơn, deterministic, không cần thêm infrastructure. Nhược điểm: không bắt được semantic similarity. Cho scope đồ án này, structured RAG đã đủ mạnh và phù hợp.
Tại sao chọn Groq mà không phải OpenAI hay Claude?
Groq cung cấp free tier đủ dùng cho development và demo. Model llama-3.3-70b có chất lượng tốt cho các task generation và classification. API tương thích với OpenAI format nên có thể migrate sang OpenAI/Claude dễ dàng nếu cần production scale.
AI có thể hallucinate không? Làm sao kiểm soát?
Có thể, nhưng em kiểm soát bằng 3 cách: (1) RAG — cung cấp data thực từ DB làm context, AI chỉ được dùng data đó; (2) Structured output — yêu cầu AI trả JSON theo format cố định, parse và validate trước khi return; (3) Fallback — nếu parse fail hoặc data rỗng, trả response default thay vì forward AI output trực tiếp. Đặc biệt AI Chatbot dùng 2-pass: pass 1 classify intent, pass 2 generate từ data thực — AI không tự bịa data.

Câu hỏi về Thanh toán

Ai trả lời: KHOA

Stripe Payment flow hoạt động cụ thể như thế nào?
4 bước: (1) Frontend gọi API tạo Payment Intent — backend gọi Stripe API, nhận client_secret; (2) Frontend hiển thị Stripe Payment Sheet với client_secret, user nhập thẻ; (3) Stripe xử lý payment và gửi Webhook event về backend endpoint /api/webhooks/stripe; (4) Backend verify webhook signature, update Payment status = COMPLETED, update Registration status = APPROVED, generate ticket code + QR. Toàn bộ card data do Stripe xử lý — backend không bao giờ thấy số thẻ.
Xử lý refund như thế nào?
Organiser nhấn Refund trên admin panel → backend gọi Stripe Refund API → Stripe hoàn tiền về thẻ user → Webhook thông báo refund success → backend update Payment status = REFUNDED, Registration status = CANCELLED. Nếu user dùng coupon, lượt dùng coupon được trả lại.

Câu hỏi về Performance & Scale

Ai trả lời: bất kỳ ai

Hệ thống có scale được không? Nếu có 10.000 users đồng thời thì sao?
Hiện tại chạy trên 1 server, phù hợp cho demo. Để scale: (1) Backend stateless (JWT) nên có thể chạy multiple instances phía sau load balancer; (2) Database có index trên các cột query thường xuyên; (3) Rate limiting bằng Bucket4j bảo vệ API; (4) Có thể thêm Redis cache cho hot data. Kiến trúc đã sẵn sàng cho horizontal scaling.
